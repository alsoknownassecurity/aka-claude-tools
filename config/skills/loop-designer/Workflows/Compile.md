# Compile — spec → minimal kickoff loop prompt

Goal: emit the SMALLEST prompt that runs the spec well in a fresh session. The orchestrator is
capable — give it the goal, the guardrails, and room. Assemble the M-core (always) + only the
N-blocks this spec truly needs. **Bias hard toward minimal; when in doubt, leave it out.**

## Step 1 — Locate the spec
`{{SPEC_PATH}}`, `{{REPO}}`. Use the spec in context if present; else read it once, fully.

## Step 2 — Fill the M-core (don't ask what the spec answers)
| Placeholder | Pull from |
|---|---|
| `{{ANTI_CRITERIA}}` | the "Anti:" criteria + constraints → the non-negotiables (one compact line) |
| `{{APPROVAL_CLAUSE}}` | set by Step 4 (the gate knob) |
The M-core needs almost nothing else filled — the spec carries the detail.

## Step 3 — Select N-blocks (sparingly)
Evaluate each N trigger (SKILL.md Tier 2). Include a block ONLY if the cost of the orchestrator
getting it wrong on THIS spec is high enough to spend a line. Defaults that usually fire: N7 (if
visual criteria), N3 (if units share files), N1 (if large), N13 (if the spec has a doctrinal
`Anti:` pattern producers could relapse on), N14 (if the loop is meant to run hands-off).
Defaults that usually DON'T: N4/N5 (a capable orchestrator handles staging/baseline) unless the
spec is unusually strict. Record each inclusion + a one-line reason for the build-sheet. If
you're unsure, omit it.

**Then sanity-check against `../FieldLessons.md`** — it captures the failures a capable
orchestrator genuinely got wrong without being told, distilled from real runs (each ⮕ GRAFT item
is a candidate block). Graft only the subset THIS spec will actually hit — still minimal, still
only what reality will punish.

## Step 4 — Confirm only what the spec can't answer
Ask the user ONLY:
1. **`{{APPROVAL}}`** — "Does launching the run count as your approval of the spec's BLOCKING
   human gates? NO (default) = it stops and asks you at each; YES = hands-off." Set
   `{{APPROVAL_CLAUSE}}` accordingly (NO → " unless I say so"; YES → add N12). If there are no
   blocking gates, skip.
2. Anything genuinely ambiguous you couldn't extract (rare). Default sensibly; state the default.

## Step 5 — Emit the prompt (minimal)
Replace `{{...}}`; append fired N-blocks under "Then:" (omit the heading if none fired). Keep it
tight — no procedure, no restating spec detail.

````
You are the orchestrator for {{SPEC_PATH}} in {{REPO}}. Drive its open criteria to verified-done
(or an explicit blocked/deferred state); loop until only human gates remain.

The spec is your source of truth — read it, derive every specific from it, and treat its
constraints and "Anti:" criteria as non-negotiable: {{ANTI_CRITERIA}}. This prompt is only the

Hold to these:
- Verify every criterion YOURSELF before marking it done — render the frame, don't trust a grep,

  never pull whole files into your own context.

You decide everything else — sequencing, what to delegate and to whom, when to parallelize,
````

## Step 5b — Persist deterministically (single file, overwritten, stamped)

The emitted block is ALSO written to ONE canonical file so a fresh session can find it. A stale
file is worse than none — so every compile **overwrites** it and **stamps its provenance**. The
staleness machinery lives in the file HEADER, *outside* the pasteable block, so the block stays
minimal.

- **Path:** `<repo-root>/EXECUTION-PROMPT.md` — a **fixed, repo-consistent location**,
  deliberately NOT co-located with the spec (spec locations vary per repo; the repo root exists
  in every repo, so the invocation is always `Read EXECUTION-PROMPT.md`). Traceability doesn't
  need co-location — the header stamp records the source spec path. **Overwrite in place** —
  never dated/versioned copies. Multi-spec repo: name a second loop
  `EXECUTION-PROMPT.<spec-stem>.md` so the two don't clobber.
- **Generated header (above the block):**
  ```
  # <Spec title> — Execution Loop Prompt
  > ⚠️ GENERATED by loop-designer from `{{SPEC_PATH}}` @ `{{SPEC_UPDATED}}` (`{{SHA}}`).
  > DO NOT hand-edit — regenerate via loop-designer. Single canonical file, overwritten each compile.
  > **STALE-IF** `{{SPEC_PATH}}` is newer than the stamp above → this prompt predates the
  > current spec. Recompile before running; never run a stale loop.
  ```
  `{{SPEC_UPDATED}}` = the spec's own `updated:`/date marker (never fabricate a separate
  timestamp); `{{SHA}}` = `git rev-parse --short HEAD`.
- Below the header: the minimal block (Step 5), then the build-sheet (Step 6).
- Commit this file in the SAME commit as the spec edits — it is a generated artifact,
  regenerated not edited.

## Step 6 — Build-sheet + launch
- **Included (and why):** the N-blocks you added, one line each; note that everything else is
  deliberately left to the orchestrator.
- **Open human gates:** BLOCKING vs TERMINAL, so the user knows where it will pause.
- **Launch:** "The compiled prompt is saved to `<repo-root>/EXECUTION-PROMPT.md` (the same path
  in every repo, overwritten this compile). From a fresh session at the repo root: `Read
  EXECUTION-PROMPT.md and execute it` — through a re-invoking loop runner if you have one (it
  adds mid-death re-entry), else as the first message of the session. Re-run loop-designer
  whenever the spec changes — the file regenerates, never hand-edit."

## Notes
- The deliverable is the emitted block AND its canonical `EXECUTION-PROMPT.md` (Step 5b) —
  written once, overwritten each compile, stamped so a clean session can detect staleness. Don't
  execute the spec here.
- The smaller the prompt, the more room the orchestrator has to react. Resist adding N-blocks
  "just in case."
